## async-generator tick ordering + return (3 tests) — splits into TWO independent sub-clusters: (1&2) explicit-return await keyed on runtime value instead of compile-time hasval [tractable]; (3) return-through-finally swallows a user throw whose OP_throw is the finally's first throw [deep, shared sync+async generator bug].

**tests**: test262/test/language/statements/async-generator/return-undefined-implicit-and-explicit.js, test262/test/language/statements/async-generator/yield-star-return-then-getter-ticks.js, test262/test/built-ins/AsyncGeneratorPrototype/return/return-suspendedYield-try-finally-throw.js
**tractability**: moderate | **files**: src/parser.zig, src/exec/async_generator.zig, src/exec/call_runtime.zig

**repro**: CLUSTER A (test 1 core): `var a=[];async function* g3(){return undefined;} async function* g1(){} Promise.resolve(0).then(()=>a.push("t1")).then(()=>a.push("t2")).then(()=>console.log(JSON.stringify(a))); g1().next().then(()=>a.push("g1")); g3().next().then(()=>a.push("g3"));` → zjs `["t1","g1","g3","t2"]` (g3 on tick1) ; qjs `["t1","g1","t2","g3"]` (g3 on tick2, because explicit return awaits).
CLUSTER A (test 2 core): the yield-star probe → zjs has THREE `get then` (`[...,"get return","get then","tick 3","get then"]`), qjs has TWO (`[...,"get return","get then","tick 3"]`).
CLUSTER B (test 3 core, reduced to SYNC generator): `var e=new Error("boop");var g=function*(){try{yield 1}finally{throw e}};var it=g();it.next();try{console.log("R:",JSON.stringify(it.return('sv')))}catch(x){console.log("THREW:",x.message)}` → zjs prints `R: {"value":"sv","done":true}` (finally throw swallowed) ; qjs prints `THREW: boop`. Contrast case that WORKS in zjs (proving it's the scan): same but `finally{try{}catch(_){}throw e}` → both throw boop.

**root cause**: TWO distinct root causes.

CLUSTER A (tests 1 + 2, tick miscount): zjs decides whether to await the async-generator return value DRIVER-SIDE at runtime, keyed on `!result.isUndefined()` (src/exec/async_generator.zig:401 execBody `.complete_result` path). qjs decides at COMPILE TIME by syntactic `hasval` and bakes OP_await into the body bytecode (emit_return quickjs.c:28401-28405: for JS_FUNC_ASYNC_GENERATOR with hasval, emit OP_await). Two failures fall out of this value-vs-syntax mismatch:
  (test 1) `return undefined` / `return void 0` are explicit returns (hasval=TRUE → qjs awaits, +1 tick) but produce runtime `undefined`, so zjs's `!result.isUndefined()` is false and skips the await → tick 2 missing between g2 and g3. Verified: `["tick 1","g1 ret","g2 ret","g3 ret","g4 ret","tick 2"]` vs qjs `[...,"g2 ret","tick 2","g3 ret",...]`.
  (test 2) yield-star's return-delegation lowering ALREADY emits OP_await in the bytecode (parser.zig:7859) mirroring emit_return(TRUE). The final return value (the thenable `{get then}`) is non-undefined, so zjs's driver-side `.complete_result` await fires AGAIN → a THIRD `get then`. Verified: zjs `[...,"get return","get then","tick 3","get then"]` vs qjs `[...,"get return","get then","tick 3"]`.
The two are the SAME bug: the runtime-value heuristic under-awaits explicit `return undefined` and double-awaits paths that already have a compiled OP_await.

CLUSTER B (test 3, finally-throw swallowed — SEPARATE, affects BOTH sync and async generators): AsyncGenerator/Generator.prototype.return() on a generator suspended at `yield` inside try/finally must run the finally; if the finally throws, .return() must reject/throw with that error. zjs reconstructs the finally range by SCANNING bytecode (call_runtime.zig:6247 findGeneratorReturnFinallyTarget → 6272 findGeneratorReturnFinallyTargetFromCatch → uses findThrowFrom(catch_target) to locate the finally's re-throw point as `stop_pc`), then re-enters the body at the finally start with stop_before_pc=stop_pc. But `findThrowFrom` returns the FIRST OP_throw at/after catch_target, which for `finally { throw err; }` IS the user's own throw. stop_before_pc then truncates execution right before that OP_throw, so the user throw never runs; the stashed pending-return value is delivered instead. Verified minimal: sync `function*(){try{yield 1}finally{throw e}}` .return('sv') → zjs resolves `{value:'sv',done:true}`, qjs throws `e`. Case E (`finally{throw new Error(...)}`) fails; case D (nested try/catch before the throw, so findThrowFrom finds an earlier desugared rethrow) accidentally works — proving the scan-heuristic is the cause. Same defect on the async path surfaces as `Test262:AsyncTestFailure:[object Object]`.

**qjs mechanism**: emit_return quickjs.c:28392 (OP_await for hasval async-generator returns at :28401-28404; OP_gosub-to-finally at :28448); js_async_generator_resume_next quickjs.c:21568 (FUNC_RET_AWAIT at :21649; end-of-body resolve at :21628-21631, no driver await); js_async_generator_completed_return quickjs.c:21530 (COMPLETED-state .return only, not the normal-return path); yield-star lowering js_parse_delegate_yield quickjs.c:28031-28130 (single emit_return(TRUE) await at :28097).

**zjs location**: src/exec/async_generator.zig:401-417 (driver `.complete_result` await, the CLUSTER A bug) + :70 enum + :737-750 trampoline; src/parser.zig:11881 & :7347 & :13077 (return_async emitters missing the async-gen OP_await) with :7859 as the already-correct yield-star precedent; src/exec/call_runtime.zig:6272-6281 findGeneratorReturnFinallyTargetFromCatch (CLUSTER B stop_pc=findThrowFrom bug) consumed at :5900-5923 (sync) and async_generator.zig:481-504 runReturnCompletion (async).

**exact fix**: CLUSTER A: (A1) add `if (s.in_async and s.in_generator) try s.emitOp(opcode.op.await);` after the return expression is parsed and before finally/iterator-close unwinding at parser.zig:11862 (main return), 7346 (emitReturnExprBranch), and 13074 (emitCapturedReturnThroughFinally, after parseExpr line 13073). (A2) delete the `!result.isUndefined()` driver await block async_generator.zig:401-417, the complete_result enum member (:70) and its trampoline arm (:737-750). CLUSTER B: stop clipping the user throw — either route .return()-through-finally via the compiled OP_gosub/OP_ret finally (bytecode.zig gosub=108/ret=109) instead of the scan, or make findGeneratorReturnFinallyTargetFromCatch's stop_pc the finally's structural end so an OP_throw inside escapes as error.JSException into the existing sync (call_runtime.zig:5920) / async (async_generator.zig:381-393) rejection paths.

**smallest win**: Cluster A (parser OP_await for explicit async-gen return + delete driver `.complete_result` await) reclaims BOTH test 1 (return-undefined) and test 2 (yield-star-return-then-getter-ticks) — 2 of the 3 tests — and is the tractable, self-contained slice touching only parser.zig return-emit sites + async_generator.zig. Cluster B (return-through-finally throw propagation) is a separate deeper slice reclaiming test 3 (return-suspendedYield-try-finally-throw) and also latently fixes the sync-generator equivalent, but it should NOT be bundled with A.

**risk**: See risk field above.

**root cause**: THREE SEPARATE causes.

(1) usage-from-eval [ENGINE gap]: `import()` from `eval("import('./x')")` resolves the specifier against cwd instead of the enclosing script's directory. In zjs, dynamicImport (src/exec/vm_eval_module.zig:119) computes the referrer as `ctx.runtime.atoms.name(function.filename)` — the *immediate* function's filename. But direct-eval compiles its body with a hardcoded `.filename = "<eval>"` (src/exec/eval_ops.zig:552-554). So `function.filename == "<eval>"`, dirname("<eval>") == ".", and `./module-code_FIXTURE.js` resolves against the runner cwd (zjs root) -> ReferenceError "could not load module filename 'module-code_FIXTURE.js'". zjs never implements GetActiveScriptOrModule's "walk past eval frames to the englobing real script" step. REPRODUCED: from zjs root with filename at the test path, zjs rejects, qjs resolves correctly.

(2) The driver: js_async_generator_resume_next (quickjs.c:21568). This is the state machine loop `for(;;)`. On resume it calls async_func_resume (21620). (3) The single await tick: js_async_generator_await (quickjs.c:21446).
